# API Function Calls — Design Spec

## Context

When xcsh interacts with REST APIs (like F5 Distributed Cloud), the LLM currently operates in
"explorer mode" — guessing endpoints, discovering response schemas, fixing parsing errors, and
iterating through trial-and-error curl calls. A simple query like "list my HTTP load balancers"
takes 6+ tool calls, ~5000 tokens, and 30-60 seconds of exploration.

The [flopach/unlocking-ai-agents-network-engineering](https://github.com/flopach/unlocking-ai-agents-network-engineering/)
repo demonstrates the alternative: pre-defined deterministic functions the LLM calls by name.
The LLM decides *what* to call; the *how* is already coded.

This design introduces a **vendor-agnostic API function-call framework** for xcsh. Vendors ship
operation catalogs as marketplace plugins. The core framework routes calls deterministically.
F5 Distributed Cloud is the first implementation, backed by the 3,234 enriched OpenAPI endpoints
in [f5xc-salesdemos/api-specs-enriched](https://github.com/f5xc-salesdemos/api-specs-enriched).

### Design Constraints

- **No MCP**: MCP injects every tool schema into context. With thousands of operations this is
  untenable. The framework exposes exactly 4 tool schemas (~600 tokens) regardless of how many
  vendor operations are installed.
- **Vendor-agnostic**: Any vendor with OpenAPI specs can generate a catalog and ship it as a
  marketplace plugin.
- **Full CRUD**: Create, read, update, delete — with danger-level safety gates.
- **Deterministic**: Pre-compiled operation definitions. No runtime API discovery or schema guessing.

## Architecture

Three layers, each with a clear boundary:

```
┌─────────────────────────────────────────────────────────┐
│  LLM Context (system prompt)                            │
│  4 tool schemas: ~600 tokens total                      │
│  api_services | api_discover | api_call | api_describe  │
└────────────────────┬────────────────────────────────────┘
                     │ tool invocation
┌────────────────────▼────────────────────────────────────┐
│  Core Framework (xcsh built-in)                         │
│  packages/coding-agent/src/tools/api-tool.ts            │
│  packages/coding-agent/src/services/api-catalog.ts      │
│  - Catalog loader: discovers installed vendor catalogs  │
│  - HTTP executor: auth injection, fetch, formatting     │
│  - Safety gate: danger-level confirmation               │
└────────────────────┬────────────────────────────────────┘
                     │ reads catalogs from
┌────────────────────▼────────────────────────────────────┐
│  Vendor Catalogs (marketplace plugins)                  │
│  marketplace/plugins/<vendor>/api-catalog.json          │
│  - Pre-compiled from vendor OpenAPI specs               │
│  - Each: operations[], auth config, base URL            │
└────────────────────┬────────────────────────────────────┘
                     │ generated by
┌────────────────────▼────────────────────────────────────┐
│  Spec Compiler (build tool)                             │
│  api-specs-enriched/scripts/compile_catalog.py          │
│  OpenAPI 3.0.3 JSON → compact catalog JSON              │
│  Reusable for any vendor with OpenAPI specs             │
└─────────────────────────────────────────────────────────┘
```

## Tool Definitions

### 1. `api_services` — List installed vendor services

**Parameters**: none

**Returns**:
```json
[
  {
    "name": "f5xc",
    "displayName": "F5 Distributed Cloud",
    "operationCount": 3234,
    "categories": ["load-balancers", "security", "dns", "..."]
  }
]
```

**Purpose**: Lets the LLM discover what API services are available in the current session.

### 2. `api_discover` — Browse available operations

**Parameters**:
| Name | Type | Required | Description |
|------|------|----------|-------------|
| service | string | yes | Vendor service name (e.g., "f5xc") |
| category | string | no | Narrow to a category (e.g., "load-balancers") |
| search | string | no | Fuzzy match on operation name/description |

**Returns**: Compact operation summaries:
```json
[
  {
    "name": "list_http_loadbalancers",
    "description": "List all HTTP load balancers in a namespace",
    "method": "GET",
    "dangerLevel": "low"
  },
  {
    "name": "create_http_loadbalancer",
    "description": "Create a new HTTP load balancer",
    "method": "POST",
    "dangerLevel": "medium"
  }
]
```

**Purpose**: Lazy-loaded catalog browsing. The full catalog is never in the system prompt.

### 3. `api_call` — Execute an operation

**Parameters**:
| Name | Type | Required | Description |
|------|------|----------|-------------|
| service | string | yes | Vendor service name |
| operation | string | yes | Operation name from catalog |
| params | object | no | Path/query parameters |
| body | object | no | Request body (POST/PUT) |

**Returns**: Structured JSON response from the vendor API.

**Behavior**:
1. Look up operation in vendor catalog
2. Resolve path template with params (e.g., `/api/config/namespaces/{namespace}/...`)
3. Apply default values from catalog (e.g., `$F5XC_NAMESPACE`)
4. Inject auth headers from vendor auth config
5. Execute HTTP request via `fetch()`
6. Return formatted response

**Safety enforcement**:
- `low`/`medium` danger: execute and report
- `high` danger: emit confirmation notice via `pushPendingAction()`
- `critical` danger: block unless explicit user override

### 4. `api_describe` — Get full operation details

**Parameters**:
| Name | Type | Required | Description |
|------|------|----------|-------------|
| service | string | yes | Vendor service name |
| operation | string | yes | Operation name |

**Returns**: Full operation detail including parameter schemas, examples, prerequisites,
common errors, and best practices:
```json
{
  "name": "create_http_loadbalancer",
  "method": "POST",
  "path": "/api/config/namespaces/{namespace}/http_loadbalancers",
  "dangerLevel": "medium",
  "parameters": [
    {"name": "namespace", "in": "path", "required": true, "type": "string", "default": "$F5XC_NAMESPACE"}
  ],
  "bodySchema": {
    "required": ["metadata.name", "metadata.namespace"],
    "properties": { "...": "..." }
  },
  "prerequisites": ["origin_pool must exist first"],
  "commonErrors": [
    {"code": 409, "reason": "Resource already exists", "solution": "Use update instead"}
  ],
  "bestPractices": ["Always create health check before origin pool"]
}
```

**Purpose**: The LLM calls this before complex operations (create/update) to understand
required fields, body shape, and prerequisites.

## Vendor Catalog Format

Each marketplace plugin ships an `api-catalog.json` at its plugin root.

### Schema

```json
{
  "service": "f5xc",
  "displayName": "F5 Distributed Cloud",
  "version": "1.0.0",
  "specSource": "f5xc-salesdemos/api-specs-enriched@v2.1.0",
  "auth": {
    "type": "api_token",
    "headerName": "Authorization",
    "headerTemplate": "APIToken {token}",
    "tokenSource": "F5XC_API_TOKEN",
    "baseUrlSource": "F5XC_API_URL"
  },
  "defaults": {
    "namespace": { "source": "F5XC_NAMESPACE" }
  },
  "categories": [
    {
      "name": "load-balancers",
      "displayName": "Load Balancers",
      "operations": [
        {
          "name": "list_http_loadbalancers",
          "description": "List all HTTP load balancers in a namespace",
          "method": "GET",
          "path": "/api/config/namespaces/{namespace}/http_loadbalancers",
          "dangerLevel": "low",
          "parameters": [
            {
              "name": "namespace",
              "in": "path",
              "required": true,
              "type": "string",
              "default": "$F5XC_NAMESPACE"
            }
          ]
        }
      ]
    }
  ]
}
```

### Auth Types

The catalog declares its auth method. The framework supports:

| Auth Type | Config Fields | Example |
|-----------|--------------|---------|
| `api_token` | headerName, headerTemplate, tokenSource | F5XC: `APIToken {token}` |
| `bearer` | tokenSource | Standard bearer auth |
| `basic` | usernameSource, passwordSource | Username/password |
| `custom` | headerName, headerValueSource | Arbitrary header |

The `tokenSource`/`baseUrlSource` fields reference environment variable names. The framework
reads values from `process.env` or from xcsh's ProfileService if available.

### Default Parameter Resolution

Parameters with `"default": "$ENV_VAR"` are auto-populated from the environment. The LLM
can omit these parameters when calling `api_call()`:

```
api_call("f5xc", "list_http_loadbalancers")
// namespace auto-resolved from $F5XC_NAMESPACE
```

## Spec Compiler

A Python script in `api-specs-enriched/scripts/compile_catalog.py` that converts enriched
OpenAPI specs into the compact catalog format.

### Input

Enriched OpenAPI 3.0.3 JSON from `docs/specifications/api/*.json`:
- `operationId` → operation name (transformed from proto-style to snake_case)
- `summary` → description
- `x-f5xc-operation-metadata.danger_level` → dangerLevel
- `x-f5xc-operation-metadata.required_fields` → bodySchema required
- `x-f5xc-operation-metadata.common_errors` → commonErrors
- `x-f5xc-best-practices` → bestPractices
- `parameters` → parameters (path, query)
- Request body schema → bodySchema

### Output

Compact `api-catalog.json` (~100-200KB for F5XC's 3,234 operations).

### Operation Name Transformation

Proto-style operation IDs are transformed to user-friendly names:
```
ves.io.schema.views.http_loadbalancer.API.List → list_http_loadbalancers
ves.io.schema.views.http_loadbalancer.API.Get  → get_http_loadbalancer
ves.io.schema.views.http_loadbalancer.API.Create → create_http_loadbalancer
```

Rule: `{verb}_{resource_plural}` for list, `{verb}_{resource_singular}` for get/create/update/delete.

## Catalog Discovery

### How the framework finds catalogs

At session start, the framework scans installed marketplace plugins for `api-catalog.json`:

```
~/.claude/plugins/marketplaces/*/plugins/*/api-catalog.json
```

Each discovered catalog is loaded and indexed by service name. The framework maintains
an in-memory registry of all available services and their operations.

### Lazy loading

Catalogs are loaded on first use, not at session start:

1. **`api_services()`**: Filesystem scan only. Checks which plugins have `api-catalog.json`
   and reads the top-level `service`, `displayName`, and category count. Does NOT parse
   operations. Cost: ~1ms per plugin.
2. **`api_discover()` / `api_call()`**: Triggers full catalog parse for the requested service.
   The parsed catalog is cached in memory for the session duration.
3. **Auth resolution**: Credentials are resolved lazily on first `api_call()` for a service.
   Fallback chain: ProfileService (if F5XC) → process.env → error with clear message.

## Safety Model

### Danger Levels

The enriched API specs include `danger_level` metadata per operation. The framework
maps these to execution behavior:

| Level | Operations | Behavior |
|-------|-----------|----------|
| `low` | list, get, describe | Execute immediately |
| `medium` | create | Execute with informational log |
| `high` | update, replace | Confirm with user via pushPendingAction |
| `critical` | delete, revoke, disable | Block unless user explicitly approves |

### Permission Model

The framework respects xcsh's existing permission system:
- Tools require user approval on first invocation (standard xcsh behavior)
- Once approved, `low`/`medium` operations run without further prompts
- `high`/`critical` operations always prompt regardless of prior approval

## Token Efficiency

| Scenario | Today (ad-hoc) | With framework |
|----------|----------------|----------------|
| System prompt cost | 0 tokens | ~600 tokens (4 tool schemas) |
| "List my LBs" | ~5000 tokens (6+ curl iterations) | ~400 tokens (1 api_call) |
| "What F5XC operations exist?" | Not possible | ~800 tokens (api_discover) |
| "Create an origin pool" | ~8000 tokens (explore + body shape) | ~600 tokens (api_describe + api_call) |
| "Delete a service policy" | ~6000 tokens (explore + confirm) | ~500 tokens (api_describe + api_call with confirm) |

**Net savings**: 80-90% fewer tokens per API operation. The 600 token upfront cost for 4 tool
schemas is amortized across every API interaction in the session.

## Testing Strategy

### Unit Tests

| Test Area | What to Test |
|-----------|-------------|
| Catalog loader | Discovers catalogs from plugin dirs, handles missing/malformed files |
| Operation lookup | Finds operations by name, category, fuzzy search |
| Path resolver | Substitutes parameters into path templates, applies defaults from env |
| Auth injector | Builds correct headers for each auth type |
| Safety gate | Blocks critical ops, prompts for high, passes low/medium |

### Integration Tests

| Test | Approach |
|------|---------|
| Full api_call flow | Mock HTTP server, verify request shape matches catalog definition |
| F5XC catalog | Load real api-catalog.json, verify all operations parse correctly |
| Error handling | 401, 403, 404, 429, 500 responses produce clear error messages |

### E2E Verification

```bash
# Start xcsh with F5XC catalog installed
# Verify:
api_services()                                    # returns f5xc
api_discover("f5xc", "load-balancers")            # returns LB operations
api_call("f5xc", "list_http_loadbalancers")       # returns real LB data
api_describe("f5xc", "create_http_loadbalancer")  # returns full schema
```

## Future Extensions

- **Batch operations**: `api_batch(service, operations[])` for parallel execution
- **Diff operations**: `api_diff(service, operation, params)` to compare current vs. desired state
- **Terraform integration**: Generate terraform resources from api_call results
- **Response caching**: Cache list/get responses within a session to reduce API calls
- **Webhook subscriptions**: Register for real-time notifications on resource changes
